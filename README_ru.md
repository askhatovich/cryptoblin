# CryptoBlin

[English](README.md) · [Русский](README_ru.md)

Сервис для самостоятельного развёртывания, реализующий обмен заметками и
файлами по модели zero-knowledge.

Сервер хранит только зашифрованные данные и не имеет доступа ни к ключу
шифрования, ни к паролю, ни к открытому содержимому. Все криптографические
операции выполняются в браузере отправителя или в CLI-клиенте `blin`;
задача сервера — выдать тот же зашифрованный блок участнику, который
доказал обладание ключом.

Название — сокращение от **Crypto Blob Insert**, одновременно совпадающее
с русским словом **блин**.

Проект продолжает идеи [PrivateBin](https://github.com/PrivateBin/PrivateBin)
и ориентирован на простоту реализации, производительность и работу с
большими файловыми вложениями.

---

## Возможности

- Полная zero-knowledge модель: сервер хранит только шифротекст и
  proof-of-key challenge с аутентификацией AEAD.
- AEAD XChaCha20-Poly1305; ключ выводится через Argon2id.
- Опциональный пароль (учитывается на стадии вывода ключа).
- Burn-after-read: запись атомарно удаляется при первом успешном открытии.
- Удаление автором по одноразовому токену с правами владельца.
- Нелинейная (экспоненциальная) зависимость срока хранения от размера:
  маленькие записи живут максимальное время, крупные — быстро истекают.
- Decoy-ответы: запрос на несуществующий id возвращает ту же форму ответа,
  что и реальный, поэтому установить наличие конкретной записи нельзя.
- C++ сервер в виде одного исполняемого файла (Crow для HTTP, SQLite для
  метаданных; libsodium используется на стороне клиентов). SPA-бандл
  встраивается в бинарь на этапе сборки.
- Кросс-платформенный CLI-клиент `blin` для Linux и Windows; поддерживается
  статическая сборка.

## Архитектура

```
                        ┌────────────────────────┐
   ┌─────────┐  https   │     cryptoblin (C++)   │
   │ browser │◀────────▶│  Crow router           │
   └────┬────┘          │  ┌──────────────────┐  │     ┌──────────────┐
        │ libsodium     │  │ SQLite metadata  │  │     │ blob files   │
        ▼ (WASM, in JS) │  └──────────────────┘  │     │ (one per id) │
   plaintext / file     │  ┌──────────────────┐  │     └──────────────┘
        │               │  │ in-memory:       │  │             ▲
        ▼ AEAD          │  │  rate-limit, PoW │  │             │
   ciphertext ──────────▶  │  replay set,     │──┼─────────────┘
                        │  │  /open sessions  │  │
                        │  └──────────────────┘  │
                        └────────────────────────┘

   key  =  Argon2id(seed + ":" + password,
                    salt = blake2b("cryptoblin/v1/" + seed))
   key  живёт только в URL-фрагменте (#…), на сервер не уходит.
```

### Формат передачи

Тело запроса `POST /api/pastes` — это один бинарный поток следующей
структуры:

```
[ 33 байта  challenge plaintext ] [32 random | 1 байт burn-флаг]
[ 24 байта  challenge nonce     ]
[ 49 байт   challenge ciphertext] (AEAD над plaintext, 16-байтный тег)
[  N байт   envelope            ] [1 ver | 24 nonce | ct + 16 mac]
```

Envelope представляет собой AEAD-обёртку над содержимым пользователя
следующего вида:

```
[ 1 байт format ] (0 = text, 1 = markdown, 2 = code)
[ u16 langLen   ][ utf-8 lang   ]
[ u32 textLen   ][ utf-8 text   ]
[ u16 nameLen   ][ utf-8 имя файла   ]
[ u16 mimeLen   ][ utf-8 mime файла  ]
[ u32 fileLen   ][ сырые байты файла ]
```

33-й байт challenge plaintext — burn-флаг — фиксируется на этапе создания
и сквозно аутентифицирован AEAD: сервер однократно читает его в открытом
виде при создании записи, а получатель, дешифровав challenge, получает
именно то значение, которое установил отправитель.

### URL-фрагменты

Маршрутизация выполняется полностью на стороне клиента. Путь HTTP-запроса
всегда `/`; вся информация о маршруте находится в URL-фрагменте, который
браузеры не отправляют на сервер:

```
/#<id>:<seed>          →  открыть, без пароля
/#<id>:<seed>:p        →  открыть, запросить пароль
/#del:<id>:<token>     →  удаление автором (выдаётся однократно при создании)
```

`id` — 8 алфавитно-цифровых символов; `seed` — 8 символов base62; `token` —
16 случайных байтов в base64.

### Формула срока хранения

Срок хранения вычисляется как экспоненциальный спад между минимальным и
максимальным значением, привязанный к концам диапазона размеров:

```
ratio = (size − 1 KiB) / (max_paste_bytes − 1 KiB)
ttl   = min_ttl + (max_ttl − min_ttl) · (e^(−5·ratio) − e^−5) / (1 − e^−5)
```

На большей части диапазона значение TTL остаётся вблизи максимума, а резкое
снижение до `min_ttl` сосредоточено у верхней границы по размеру. При
значениях по умолчанию (30 дней / 1 час и лимите 100 МиБ):

| размер     | TTL          |
|------------|--------------|
| ≤ 1 КиБ    | 30 дней      |
| 10 МиБ     | ~ 18 дней    |
| 50 МиБ     | ~ 2.3 дня    |
| 90 МиБ     | ~ 4.2 часа   |
| 100 МиБ    | 1 час        |

## API

Все ответы — JSON, если не оговорено отдельно. Ошибки имеют форму
`{ "error_code": "<код>", "message": "<опц.>" }` со статусом 4xx/5xx.

| Метод   | Путь                        | Назначение                                           |
|---------|-----------------------------|------------------------------------------------------|
| `GET`   | `/api/config`               | имя и заголовок приложения, версия, лимиты           |
| `GET`   | `/api/captcha`              | выдать новый подписанный PoW-токен                   |
| `POST`  | `/api/pastes`               | создать запись (бинарное тело типа octet-stream)     |
| `POST`  | `/api/pastes/{id}/open`     | получить challenge (`nonce` + `ciphertext`)          |
| `POST`  | `/api/pastes/{id}/blob`     | предъявить расшифрованный plaintext и получить blob  |
| `DELETE`| `/api/pastes/{id}`          | удаление автором (заголовок `X-Delete-Token`)        |

### Поток создания

1. Клиент запрашивает `/api/captcha` → `{ token, difficulty }`.
2. Клиент перебирает nonce так, чтобы
   `leading_zero_bits(SHA-256(token + ":" + nonce)) ≥ difficulty`.
3. Клиент формирует бинарное сообщение и отправляет `POST /api/pastes` с
   заголовками `X-Pow-Token` и `X-Pow-Nonce`. Сервер повторно проверяет
   токен и помечает его как использованный (одноразовое множество для
   защиты от повторного использования), считывает burn-флаг из 32-го байта
   plaintext, записывает строку в БД и blob на диск.
4. Сервер возвращает `{ id, expires_at, size, ttl_seconds, burn, delete_token }`.

### Поток открытия

1. `POST /api/pastes/{id}/open` →
   `{ token, challenge: { nonce, ciphertext } }`. Для несуществующих id
   возвращается синтетический decoy с равномерно случайными байтами той же
   формы; никаких метаданных на этом шаге не передаётся.
2. Клиент расшифровывает challenge своим ключом, извлекает proof-plaintext
   и читает burn-флаг — он используется, чтобы показать пользователю
   предупреждение «запись будет уничтожена» до запроса blob.
3. `POST /api/pastes/{id}/blob` с телом `{ token, plaintext: <b64> }`.
   Сервер сравнивает значение с сохранённым plaintext с постоянным временем
   выполнения. При совпадении blob передаётся потоком в ответе; при
   `burn=true` строка БД атомарно удаляется до отправки тела. Счётчик
   просмотров и флаг `burned` возвращаются в заголовках ответа.

### Поток удаления

`DELETE /api/pastes/{id}` с заголовком `X-Delete-Token: <b64>`. Сервер
хранит SHA-256 от токена; в DB-слое сначала читается сохранённый хэш и
сравнивается с присланным с постоянным временем выполнения, затем при
совпадении выполняется удаление строки. Отсутствие токена, неверный токен
и отсутствие id приводят к одному и тому же ответу `403 bad_token` без
наблюдаемых временны́х различий.

## Модель угроз

CryptoBlin защищает определённые данные от определённых классов
атакующих. Перечисленные ниже ограничения существенны и должны
учитываться при развёртывании.

### Что закрывается

- **Любопытный, но честный администратор сервера.** На сервере хранятся
  шифротекст, случайная соль, непрозрачный AEAD-challenge и SHA-256 от
  delete-токена. Чтение диска не даёт plaintext. Серверной копии ключа в
  системе не существует.
- **Кража БД или резервной копии.** Аналогично: похищенная база раскрывает
  размеры записей и временные метки, но не их содержимое.
- **TLS-терминатор или reverse-proxy.** Ключ шифрования не попадает в
  HTTP-трафик. Он находится только в URL-фрагменте, который браузеры не
  отправляют на сервер.
- **Пассивный сетевой наблюдатель.** Шифр — XChaCha20-Poly1305; ключ
  выводится из seed (и опционального пароля) через Argon2id с лимитами
  `OPSLIMIT_INTERACTIVE` и `MEMLIMIT_INTERACTIVE`. Без HTTPS наблюдатель
  не может прочесть содержимое; при использовании HTTPS он также не видит
  идентификаторы и размеры записей.
- **Подмена шифротекста сервером.** AEAD аутентифицирует envelope.
  Burn-флаг дополнительно зафиксирован внутри AEAD-plaintext, поэтому
  сервер не может незаметно переключить значение `burn` для записей,
  которые он не способен расшифровать.
- **Проверка существования id.** `/api/pastes/{id}/open` возвращает
  одинаковую JSON-форму для реального и отсутствующего идентификатора.
  `/blob` сводит все режимы отказа к ответу `403 bad_key`. Decoy-challenge
  заполняется равномерно случайными байтами.
- **Гонка burn-after-read.** Burn реализован атомарным
  `DELETE … WHERE id = ?`; запись захватывается только одним вызовом
  `/blob`, остальные получают `bad_key`.
- **Повторное использование captcha и злоупотребление ей.** PoW-токен
  подписан HMAC, одноразовый, с ограниченным временем жизни. Выпуск
  captcha ограничен по IP, чтобы клиент не мог исчерпать множество
  использованных токенов.
- **Случайное срабатывание burn.** CLI и веб-клиент расшифровывают
  burn-флаг **до** обращения к `/blob`; если он установлен, пользователю
  показывается подтверждение, что запись будет уничтожена при продолжении.

### Что не закрывается

- **Скомпрометированная конечная точка.** Кейлоггер, вредоносное
  расширение браузера или подменённый исполняемый файл `blin` на одной из
  сторон обходят модель. Проверяйте SHA-256 артефактов релизов (GitHub
  публикует их на странице релиза).
- **Активный MITM без HTTPS.** Атакующий, способный перехватить и
  подменить отдаваемый HTML, заменит бандл и получит URL-фрагмент.
  В рабочей среде использование HTTPS обязательно.
- **Перебор слабых паролей.** Argon2id настроен с интерактивными лимитами
  стоимости; короткие пароли всё ещё поддаются перебору. Используйте
  достаточно сложные пароли или полагайтесь только на ключ из URL.
- **Утечка или повтор полного URL.** Любой, кто получает URL целиком
  (включая фрагмент), сможет прочитать запись до истечения её срока или
  до срабатывания burn. URL следует обращаться как с секретом.
- **Анонимность и метаданные.** Сервер по умолчанию записывает IP-адреса
  для rate-limit и выдачи PoW. CryptoBlin защищает содержимое, а не
  обеспечивает анонимность; для последней используйте Tor или прокси.
- **Отказ в обслуживании.** PoW, ограничение по IP и лимит на размер
  записи повышают стоимость массового создания, но атакующий, обладающий
  большим пулом IP-адресов, всё ещё способен перегружать чтения или
  исчерпывать общую квоту записей. Для прикладных лимитов на уровне L7
  следует размещать перед сервисом штатный HTTP-фронтенд (nginx, Caddy и
  т. п.).

## Сборка

### Сервер (Debian 12+ / Ubuntu LTS)

```sh
sudo apt install build-essential cmake pkg-config \
                 libsqlite3-dev libsodium-dev libmbedtls-dev libasio-dev \
                 nodejs npm

cd web && npm ci && npm run build && bash embed.sh && cd ..
cmake -B build -DCMAKE_BUILD_TYPE=Release
cmake --build build -j
./build/src/cryptoblin --config /etc/cryptoblin.ini
```

`cmake --install build` устанавливает `cryptoblin` и `blin` в префикс.

### Только CLI (`blin`)

```sh
cmake -B build -DBUILD_SERVER=OFF -DBUILD_CLI=ON
cmake --build build -j
```

### CLI под Windows

В CI бинарь `blin.exe` собирается через vcpkg со статической линковкой
зависимостей (libsodium, mbedTLS); CRT остаётся динамической. Локальная
сборка:

```pwsh
vcpkg install libsodium:x64-windows-static-md mbedtls:x64-windows-static-md
cmake -B build -DBUILD_SERVER=OFF -DBUILD_CLI=ON `
  -DCMAKE_TOOLCHAIN_FILE="$env:VCPKG_INSTALLATION_ROOT/scripts/buildsystems/vcpkg.cmake" `
  -DVCPKG_TARGET_TRIPLET=x64-windows-static-md
cmake --build build --config Release -j
```

## Конфигурация (`config.ini`)

```ini
[server]
log_level    = info
bind_address = 0.0.0.0
bind_port    = 8080
app_name     = CryptoBlin
title        =                 # опц. переопределение для хедера и <title>

[storage]
db_path  = /var/lib/cryptoblin/cryptoblin.sqlite
blob_dir = /var/lib/cryptoblin/blobs

[retention]
max_paste_bytes = 104857600    # 100 МиБ
min_ttl_seconds = 3600
max_ttl_seconds = 2592000

[limits]
rate_limit_per_ip_seconds = 3
total_pastes_cap          = 0  # 0 = без ограничения

[captcha]
difficulty_bits = 18
ttl_seconds     = 300
```

## CLI

```sh
echo "secret" | blin send                        # текст со stdin
blin send -b README.md                            # burn-after-read, только файл
echo "see attached" | blin send report.pdf       # текст и вложение
blin get https://paste.dotcpp.ru/#abcd1234:wxyz5678
blin delete https://paste.dotcpp.ru/#del:abcd1234:AAA…
```

URL сервера читается из `$HOME/.config/askhatovich/cryptoblin/config.conf`
(формат: `SERVER=https://example.com`); по умолчанию используется
`https://paste.dotcpp.ru`. Для команд `get` и `delete` CLI использует
сервер из самой ссылки, если она передана с префиксом `http(s)://host/`;
файл конфигурации в этом случае служит резервным значением.

`blin --help` описывает все флаги и контракт между stdout и stderr:
полезные данные направляются в stdout, записи `key: value` и
информационные сообщения — в stderr.

## Разработка

```sh
# модульные тесты сервера
ctest --test-dir build --output-on-failure

# end-to-end тесты против запущенного экземпляра сервера
cd web && node --test tests/
```

Релизы собираются через GitHub Actions при публикации тегов `v*`:
`.deb`-пакеты сервера для Debian 12 и Debian 13, tar-архив со статически
слинкованным под musl `blin` для Linux и zip-архив с `blin.exe` для
Windows. GitHub автоматически публикует SHA-256 для каждого артефакта на
странице релиза.
